1 Writing & Content

9 prompts — blogs, email, social, careers, and narrative marketing

Blog post generator

Act as a senior editor and SEO-aware writer. Write a blog post on [TOPIC] for [AUDIENCE — e.g. startup founders]. Goals: rank for intent around [PRIMARY KEYWORD] without keyword stuffing. Structure: (1) headline + subhead promise, (2) hook in the first 120 words, (3) 4–6 H2 sections each ending with a takeaway, (4) one numbered “quick checklist”, (5) FAQ with 4 Q&As, (6) conclusion with one clear CTA. Tone: [professional / conversational]. Length: ~[N] words. Use specific examples; flag any claims that need a citation as [VERIFY].

Give 3 competitor article titles you want to beat — Claude will sharpen angles.

Professional email

Draft a professional email. To: [ROLE / NAME]. Subject: [THREAD TOPIC]. My goal: [e.g. request a 30-min call next week]. Context (facts only): [PASTE 3–6 BULLETS OR SUMMARY]. Constraints: max [N] words, tone [warm-neutral / firm / apologetic], avoid jargon. Output: Subject line + body. If declining or pushing back, offer one constructive alternative. If following up, reference prior touchpoint without sounding passive-aggressive.

Paste the last email you received (redacted) so the reply matches thread tone.

Social media captions

Create social captions for [PLATFORM: LinkedIn / X / Instagram]. Topic: [ONE-SENTENCE HOOK]. Brand voice: [3 adjectives]. Deliver: (1) 1 long post (with line breaks), (2) 3 short variants under 280 chars, (3) 10 hashtag ideas (mix broad + niche), (4) 2 CTA options. Rules: no engagement bait, no fake urgency, no unverifiable stats. Include one optional “thread starter” reply prompt for comments.

Add “banned phrases: [list]” to keep voice on-brand.

Product descriptions

Write ecommerce product copy for [PRODUCT NAME]. Audience: [WHO BUYS IT]. Specs / facts: [PASTE BULLETS]. Differentiators vs alternatives: [2–4 BULLETS]. Output: (1) title (≤70 chars), (2) short description (1–2 sentences), (3) benefits-led bullets (5), (4) “what’s in the box”, (5) care / compatibility notes if relevant. Style: scannable, confident, no superlatives without proof. Flag compliance-sensitive claims as [CLAIM — NEEDS PROOF].

Include return policy and shipping region — Claude can weave trust without legal overreach.

Resume bullet points

Rewrite my resume bullets to be achievement-focused for [ROLE YOU’RE TARGETING]. For each bullet I paste, apply: Action verb + scope + metric + impact + tool/method (if true). Rules: don’t invent metrics; if unknown, use [METRIC TBD] and suggest what to measure. Keep each line ≤ 28 words. Output a table: Original | Improved | Why it’s stronger. Cover letter

Requirements: 250–350 words, 3 paragraphs, no clichés (“passionate”, “synergy”), include one concrete project story with outcome, connect my skills to their stated priorities, end with a confident close (not desperate). If company values/mission are known: [PASTE], align subtly once.

Name the hiring manager if known — personalization beats generic openers.

Newsletter

Draft a newsletter issue for [AUDIENCE]. Theme: [THEME]. Sections required: Subject line (2 options) + preview text, Intro (100–140 words), “3 things” curated list with 2-sentence blurbs + links placeholders [LINK], One original mini-essay (250–350 words), Tool/resource spotlight, Closing + P.S. Tone: [voice guidelines]. Constraints: reading time ~[N] min, skimmable headings, one clear CTA. If I paste links later, don’t hallucinate summaries—use placeholders.

Specify send day (“Sunday brief”) so pacing and CTA match reader context.

LinkedIn post

Write a LinkedIn post for [YOUR ROLE/INDUSTRY]. Core idea: [ONE PARAGRAPH]. Format: hook line, short story or framework, 3 numbered lessons, ending question. Use whitespace and short paragraphs. Avoid: “Agree?”, “Thoughts?”, buzzword stacking. Optional: add a simple ASCII or bullet framework. Length: 1,200–1,800 characters. Include 3 comment prompts that invite expertise, not platitudes.

Add “controversy level: low/medium” to calibrate how sharp the take sounds.

Results: [metrics — mark unknowns as TBD]. 2 Coding & Technical

9 prompts — debug, tests, APIs, SQL, reviews, and translation

Debug code

You are a staff engineer. Debug the following code. Steps: (1) restate the expected vs actual behavior, (2) list hypotheses ordered by likelihood, (3) pinpoint root cause with line references, (4) propose minimal fix with code diff, (5) add 2–3 tests or assertions that would have caught this, (6) note any security/perf footguns. If info is missing, ask up to 3 targeted questions first. Stack: [LANGUAGE/FRAMEWORK]. Error/logs: [PASTE]. Code: [PASTE].

Include inputs that reproduce the bug — fewer rounds of back-and-forth.

Refactor code

Refactor for readability and maintainability without changing behavior. Constraints: preserve public API, no new dependencies unless justified, prefer small pure functions, add types if TypeScript. Deliver: (1) refactored code, (2) brief rationale for structural changes, (3) “before/after” complexity notes, (4) follow-up checklist (tests to run). If you must break behavior, STOP and explain. Code: [PASTE]. Context: [WHAT THIS MODULE DOES].

Say “max cyclomatic complexity 10 per function” for tighter decomposition.

Write unit tests

Generate unit tests using [Jest / Vitest / pytest / Go test]. Target: [FILE/FUNCTION]. Requirements: cover happy path, edge cases, and failure modes; use table-driven tests where appropriate; mock external I/O; avoid flaky timing. Output: complete test file + commands to run. If coverage gaps remain, list them explicitly. Code under test: [PASTE].

Paste your testing conventions (e.g. Arrange-Act-Assert) for consistent style.

Explain complex code

Explain this code like I’m a [junior/mid/senior] [LANGUAGE] developer but new to this codebase. Output: (1) one-paragraph purpose, (2) step-by-step execution trace for typical input, (3) diagram in text (flow or sequence), (4) glossary of non-obvious terms, (5) “gotchas” and invariants, (6) 5 quiz questions to check understanding. Code: [PASTE]. Entry point / call site: [IF KNOWN].

Ask for “mental model first, details second” when you’re overwhelmed by line noise.

API endpoint generator

Design a REST (or RPC) API for [RESOURCE]. Framework: [Express / FastAPI / Next route handlers / etc.]. Requirements: auth model [JWT/session/API key], idempotency for writes if needed, pagination style, error schema, rate-limit notes. Deliver: OpenAPI-style paths table, example request/response JSON for 200/400/401/404/409/422/500, and stub handler code with validation. Data model: [PASTE OR DESCRIBE].

Mention your DB (Postgres vs Dynamo) — shapes and constraints will match reality.

SQL query builder

Write SQL for [Postgres / MySQL / SQLite / BigQuery]. Task: [DESCRIBE REPORT OR UPDATE]. Schema: [PASTE DDL OR DESCRIBE TABLES]. Rules: parameterize user inputs in comments, avoid SELECT *, add appropriate indexes suggestions (CREATE INDEX …) if slow paths likely, use CTEs for readability. Output: final query + brief explanation + sample output rows (fictional ok). If ambiguous, propose two interpretations.

Paste EXPLAIN ANALYZE output for tuning — Claude can suggest concrete rewrites.

Code review

Perform a PR review on the diff below. Categories: correctness, security, performance, API compatibility, tests, observability, naming/docs. Use severity tags: [BLOCKER] [MAJOR] [MINOR] [NIT]. For each issue: location, why it matters, suggested fix. End with a merge recommendation: approve / approve with nits / request changes. Diff / files: [PASTE]. Context: [FEATURE GOAL]. Threat model notes: [IF ANY].

Include your team’s standards link or bullet list so feedback aligns with house style.

Language conversion

Translate this implementation from [LANG A] to [LANG B]. Preserve behavior and public interfaces. Map idioms (async, errors, null handling). Deliver: (1) translated code, (2) dependency mapping table, (3) behavioral parity checklist, (4) tests ported or rewritten. Flag any semantic gaps (e.g., integer size, timezone). Source: [PASTE].

Note target version (e.g. Python 3.12, Java 21) to avoid deprecated APIs.

CI-friendly test plan (bonus)

Given this feature spec: [PASTE]. Produce a CI-friendly test plan: unit/integration/e2e split, data fixtures, flaky-risk areas, feature flags, rollback checks. Output: bullet test cases with Given/When/Then, priority (P0–P2), and which layer runs in PR vs nightly. Include minimal mocking strategy and observability assertions (logs/metrics) if relevant.

Attach your pipeline tool (GitHub Actions, etc.) for copy-paste workflow stubs.

3 Business & Productivity

8 prompts — meetings, strategy, planning, and stakeholder comms

Meeting summary

Summarize this meeting for stakeholders who weren’t present. Input: raw notes or transcript [PASTE]. Output: (1) 5-bullet executive summary, (2) decisions made (owner + deadline), (3) open questions, (4) risks/blockers, (5) next steps as a checklist. Rules: attribute decisions only when explicit; otherwise mark as [UNCONFIRMED]. Tone: neutral, factual. Redact PII as [REDACTED].

Paste attendee list so action owners are assigned to real roles.

SWOT analysis

Create a SWOT for [COMPANY / PRODUCT / INITIATIVE]. Context: market, customers, competitors, capabilities [PASTE]. Requirements: 4–5 bullets per quadrant, each bullet starts with a bold label, include “so what” implication for top 2 items per quadrant, note data gaps as [ASSUMPTION]. Avoid generic platitudes—tie strengths/weaknesses to evidence. End with 3 strategic moves: mitigate weakness, exploit opportunity, monitor threat.

Add your moat and distribution reality — SWOTs fail when strengths are wishful.

Project plan

Build a project plan for [GOAL]. Constraints: budget [X], timeline [Y], team [ROLES], tools [STACK]. Deliver: milestones (with dates), work breakdown (2-level), dependencies, RACI for key deliverables, risk register (probability/impact + mitigation), communication cadence, definition of done. Use a table for milestones. Flag unknown estimates as [TBD] with what data would de-risk them.

Name your release strategy (big-bang vs iterative) — plans change a lot based on it.

Competitor analysis

Analyze competitors for [CATEGORY]. Competitors: [A, B, C]. My product: [ONE-PARAGRAPH]. Output: comparison table (pricing motion, ICP, core features, integrations, positioning, strengths, weaknesses, likely next moves). Add “watch list” signals (hiring, releases, pricing). Mark uncertain items [INFERRED]. Finish with 5 positioning angles we can credibly own and 3 landmines to avoid.

Paste their homepage hero copy — positioning gaps surface faster than feature lists.

Pitch deck outline

Outline a [N]-slide pitch deck for [STAGE: pre-seed/seed/Series A] [COMPANY]. Audience: [investors/partners]. Facts to preserve: [TRACTION, TEAM, MARKET, PRODUCT — PASTE]. For each slide: title, 3–5 bullets max, speaker notes (30–45s), and one visual suggestion. Include expected investor objections and where to preempt them. Avoid hype; quantify claims or tag [NEEDS DATA].

Specify round size and use of funds — slide order shifts when the ask is crisp.

OKR generator

Draft quarterly OKRs for [TEAM] aligned to company theme: [THEME]. Inputs: current metrics [PASTE], constraints [PASTE]. Rules: 3 Objectives max, 2–4 KRs each, every KR has baseline, target, and measurement method + data source. Use outcome KRs, not task lists. Include a “health metrics” section (quality, reliability, debt) that must not regress. Flag conflicts or dependencies between KRs.

Paste last quarter’s misses — good OKRs encode honest baselines.

Status report

Write a weekly status report for [PROGRAM]. Audience: [exec/engineering/customer]. Sections: headline (1 sentence), shipped, in progress (% confidence + ETA), blocked (owner + ask), metrics (table), customer/stakeholder feedback themes, risks (severity), next week plan. Tone: crisp, no blame. If I paste raw updates, normalize voice and remove duplicates. Redact sensitive names if marked.

Add “max 1 page” — forces signal over activity theater.

Process documentation

Document an operational process for [PROCESS NAME] so a new hire can run it. Inputs: rough steps [PASTE]. Output: purpose, scope, roles, prerequisites, step-by-step SOP (numbered), decision tree for common exceptions, SLAs, tooling links as [LINK], audit trail expectations, and failure modes + escalation. Add “failure story from last month” — exceptions become first-class in the SOP.

4 Research & Analysis

6 prompts — synthesis, evidence, and decision-ready views

Literature review

Produce a structured literature review on [RESEARCH QUESTION]. Sources I provide: [PASTE TITLES/ABSTRACTS/LINKS]. Output: (1) scope and inclusion criteria, (2) thematic synthesis (not article-by-article laundry list), (3) comparison of methods/findings, (4) gaps and contradictions, (5) implications, (6) suggested experiments/next studies. Mark where you lack full text as [NOT VERIFIED FROM SOURCE]. Academic tone, precise verbs, no fabricated citations.

Upload PDF excerpts you’re allowed to share — summaries stay grounded.

Data interpretation

Interpret the following dataset/results for a [AUDIENCE: exec/data team]. Data: [PASTE TABLE OR DESCRIBE]. Questions: [LIST]. Deliver: key takeaways (3–5), charts to consider (describe each), statistical cautions (sample size, confounders, seasonality), alternative explanations, and recommended decisions with confidence levels (high/med/low). If you cannot justify a claim from the data, say so explicitly.

State how the data was collected — interpretation without provenance misleads.

Market research

Create a market research brief for [PRODUCT/SERVICE] in [GEO/SEGMENT]. Sections: problem definition, TAM/SAM/SOM methodology (assumptions explicit), customer segments + jobs-to-be-done, buying process + budget owners, competitive alternatives (including “do nothing”), pricing benchmarks, distribution channels, regulatory notes if relevant. Use tables. Tag uncertain numbers [ESTIMATE] and list what primary research would validate them.

Name your ICP narrowly — “SMB” is too vague for useful market math.

Trend analysis

Analyze trends impacting [INDUSTRY] over [TIMEFRAME]. Inputs: signals I provide [PASTE LINKS/NOTES]. Output: trend map (macro → industry → company implications), second-order effects, early vs late signals, scenarios (base/upside/downside), and a prioritized watchlist with metrics to track. Separate fad vs durable shift. If evidence is thin, label speculation clearly.

Ask for “2026–2028 implications” to force forward-looking specificity.

Comparison table

Build a decision comparison table for choosing between [OPTION A] and [OPTION B] (add C if needed) for [USE CASE]. Criteria weights: [LIST + WEIGHTS SUM 100]. For each cell: score 1–5 with a one-sentence rationale. Include cost, time-to-value, risk, compliance, team skill fit, vendor lock-in, and maintainability. End with a recommendation and the top 3 uncertainties that would flip the decision. Avoid ties—force a justified choice.

Paste your non-negotiables first — weights should reflect real constraints.

Executive summary

Write a 1-page executive summary from the following long material [PASTE]. Audience: [ROLE]. Goal: [DECISION REQUESTED]. Format: context, key findings (bullets), risks, options (pro/con), recommendation, next steps with owners/dates. Constraints: max ~350 words, no jargon without definition, every number must appear in source or be labeled [NEW ESTIMATE]. Tone: confident, calm.

Say “assume reader has 60 seconds” — you’ll get tighter framing.

6 Power Commands

8 prompts — reliability, voice, personas, depth, and structure shortcuts

L99 (commit to an answer)

L99: You must produce a single best answer now—no “it depends” without resolving it. If information is missing, state the single most reasonable assumption in one line, label it ASSUMPTION, then proceed. Deliver: final recommendation first, then rationale in bullets, then risks/mitigations. Do not ask clarifying questions unless output would be unsafe or unethical without them.

Use when you’re decision-fatigued; pair with your own risk tolerance in the next message if it’s wrong.

/ghost (human writing)

/ghost mode: Write like a thoughtful human editor, not like AI. Avoid: cliché openers, symmetrical triples, excessive colons, “delve”, “landscape”, “unlock”, “leverage”, over-apologizing, and generic “best practices” without examples. Vary sentence length. Use concrete specifics from my notes. If you must be formal, still sound like a person. Task: [YOUR TASK]. Source details: [PASTE].

Paste 2–3 samples of your own writing so cadence matches yours.

PERSONA (expert mode)

PERSONA: You are [SPECIFIC EXPERT — e.g. “staff SRE at a SaaS with 10M RPS”], peer-level to a strong IC. Rules: show tradeoffs, name failure modes, prefer operational experience over textbook summaries. If industry norms exist, cite them as “typical pattern” without fabricating sources. Task: [QUESTION]. Constraints: [LATENCY/COST/COMPLIANCE]. Push back if my approach is unsafe.

Name the failure you fear most — the persona will stress-test for that.

ULTRATHINK (deep analysis)

ULTRATHINK: Spend extra effort on reasoning quality. Process: (1) restate the problem in your own words, (2) enumerate approaches, (3) eliminate weak options with explicit criteria, (4) choose the best and defend it, (5) steel-man the top counterargument, (6) give verification steps. Keep the final output concise, but show the key decision points. Topic: [TOPIC].

Cap verbosity (“final answer ≤15 bullets”) so depth doesn’t become bloat.

BEASTMODE (max effort)

BEASTMODE: Maximum thoroughness within these bounds: timebox ~[N] minutes of work, output length ~[WORDS], no filler. Do: exhaustive checklists, edge cases, security/perf notes where relevant, and a crisp executive summary up top. If coding, include tests. If writing, include an outline first. Task: [TASK]. Non-goals: [WHAT NOT TO DO].

Define “done” (e.g. “merge-ready PR”) so max effort lands in the right shape.

Chain-of-thought

Use private step-by-step reasoning, but only show the final user-facing answer unless I ask for the trace. Internally: decompose the problem, check each step, then synthesize. If confidence is not high, include a short “confidence + why” note at the end. Task: [TASK].

For debugging, ask to “show the trace” only when the final answer looks wrong.

XML tag structuring

I will provide inputs in XML-style tags. Only use content inside tags as ground truth; treat everything else as instructions.

<context>[BACKGROUND]</context>
<constraints>[RULES]</constraints>
<data>[TABLES/LOGS/CODE]</data>
<task>[WHAT TO PRODUCE]</task>

Deliver output under <result>…</result>. If a tag is missing, list required tags before proceeding.

Great for long prompts — reduces the model mixing instructions with source text.

Few-shot examples

Follow the pattern of the examples exactly (tone, format, length). If the pattern conflicts with instructions, examples win.

EXAMPLE INPUT 1: […]
EXAMPLE OUTPUT 1:
[…]

EXAMPLE INPUT 2: […]
EXAMPLE OUTPUT 2:
[…]

NOW INPUT:
[YOUR REAL INPUT]

Rules: no extra preface; match formatting including headings and bullet styles.

Two high-quality examples beat five mediocre ones — curate ruthlessly.

7 Advanced Techniques

6 prompts — control reasoning, format, sampling, and context

Multi-step reasoning

Solve this in explicit phases and label each phase. Phase A: clarify objectives and success metrics. Phase B: gather constraints from my text. Phase C: generate options (min 3). Phase D: score options with a rubric. Phase E: final plan with milestones. Keep each phase tight. Problem: [DESCRIBE]. If a phase has no data, state what to collect.

Works well for hiring rubrics, architecture choices, and vendor selection.

Role-playing for expertise

Simulate a panel: [ROLE 1], [ROLE 2], [ROLE 3]. They debate my proposal: [PROPOSAL]. Each speaker: 1 critique + 1 improvement. Then a chairperson synthesizes a single recommended path. Rules: disagree realistically; no fake citations. If something needs legal/finance/pro medical review, flag REVIEW REQUIRED.

Pick roles that naturally conflict (e.g. security vs growth) for sharper tension.

Constraint specification

Hard constraints (must never violate): [LIST]. Soft constraints (optimize if possible): [LIST]. Optimization priority order: [P1, P2, P3]. Tradeoff rule: if P1 conflicts with P2, choose P1 and document the cost. Task: [TASK]. Output must include a “constraint checklist” proving compliance.

Separating hard/soft prevents the model from “negotiating” legal or safety limits.

Output formatting

Output format requirements: use Markdown; top-level sections exactly: ## Summary, ## Details, ## Next Actions; use tables for comparisons; code in fenced blocks with language tags; max 12 bullets in Summary. Do not include a preamble. Task: [TASK]. If you cannot satisfy format, output ## Summary with the blocker only.

Paste into tools that consume Markdown — consistent headings make automation easy.

Temperature control tips

(Use in apps that expose sampling controls.) Task type: [CODE / LEGAL-LIKE / CREATIVE BRAINSTORM / DATA EXTRACTION]. Recommend settings reasoning: lower randomness for extraction, formatting, and codegen; higher only for divergent ideation. If you can’t set temperature: instruct “deterministic listing” vs “10 wild ideas” in natural language. My task: [TASK] — give me two variants: conservative and exploratory.

When temperature isn’t exposed, duplicate the task: first “strict/factual”, then “creative options”.

Context window optimization

I will paste a long document. First, return a structured outline with pointers (“section → key claims”). Then ask me which sections matter for: [GOAL]. After I answer, only then produce the final deliverable using those sections; ignore the rest unless contradictions appear. If the doc is noisy, propose a 10-line “source of truth” extract I should confirm. DOCUMENT: [PASTE].

For huge PDFs, chunk by chapter and reuse the outline step each time.
